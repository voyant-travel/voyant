# Voyant Agent Workflow

This is the operational policy agents and runners load before working in this
repo. `AGENTS.md` is the short map; this file is the execution contract.

## Work Intake

Agents may work only from maintainer-approved tasks.

Required gate:

- label: `agent:ready`
- Project field: `Agent State = Ready`
- Project field: `Maintainer Approved = Yes`

Triage agents may classify issues, draft briefs, and recommend next state. They
must not set the final execution gate.

## Branches And PRs

Use one branch per task.

Branch format:

```text
<type>/<issue-number>-<short-slug>
```

Examples:

- `fix/123-calendar-date-filter`
- `feature/456-agent-runner`
- `docs/789-agent-workflow-policy`
- `chore/321-quality-checker`

Do not name branches after the agent or provider. Avoid `codex/*`,
`claude/*`, or similar provider names.

PR titles, PR descriptions, commit messages, and contributor metadata should
describe the work. Do not add provider signatures, "generated by" notes,
co-author trailers, or contributor entries naming an AI provider unless a
maintainer explicitly asks for provenance disclosure.

## Required Handoff

Every task ends with an evidence packet:

- summary of changes
- files touched
- verification commands and outcomes
- screenshots or videos for UI changes
- security notes when the task touches a sensitive area
- known residual risks
- links to PR, CI run, and artifacts
- explicit handoff state

The runner should reject handoff when evidence is missing or verification failed
without an accepted reason.

## Quality Rules

- Keep routes thin: parse input, resolve services, call domain workflows, and
  serialize responses.
- Use `parseJsonBody(...)` and `parseQuery(...)` for new package routes.
- Do not add in-process tenant scoping to `packages/*`.
- Prefer typed SDK/RPC/client surfaces for reused cross-boundary calls.
- UI code should prefer existing `*-react` hooks over ad hoc fetch/state code.
- Reuse `packages/ui`, package-specific `*-ui` packages, and existing template
  components before creating local UI.
- Wire user-facing UI copy through the existing i18n system.
- Add Changesets when public package behavior, exports, styles, assets, or docs
  that ship with packages change.
- Generate database migrations through Drizzle tooling by default.
- Stop and report blockers before inventing manual migrations.
- Keep files small: warn above 500 lines and fail above 600 lines unless an
  exception is documented.
- Avoid `@ts-ignore`, `@ts-nocheck`, `as unknown as`, repeated `as any`, and
  blanket lint disables unless the exception is local, justified, and owned.
- Prefer deep modules with small interfaces and meaningful behavior behind them.
- Name files by domain behavior, not implementation trivia.

## Verification

Use the smallest lane that matches risk:

- focused package lane: `pnpm --filter <package> typecheck`, `test`, or `lint`
- fast repo lane: `pnpm verify:fast`
- full repo lane: `pnpm verify:full`

Agent-specific quality feedback:

```bash
pnpm verify:agent-quality:changed
```

This lane starts in reporting mode. Treat findings as review feedback unless
the runner or maintainer marks them blocking.

## Plans

Use `docs/agent-plans/active/` for multi-hour work. The plan must be good
enough for a fresh agent to resume after context loss. Move completed plans to
`docs/agent-plans/completed/` and abandoned plans to
`docs/agent-plans/abandoned/`.
