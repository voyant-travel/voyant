Closes #502: locale-aware product category & tag names for the catalog plane.

Adds `product_category_translations` and `product_tag_translations` tables, mirroring the existing `destination_translations` shape. The catalog plane's taxonomy projection (PR2 of #493) now reads slice-locale rows and falls back to the canonical `productCategories.name` / `productTags.name` when no translation exists for a given locale.

Schema:

- `product_category_translations` (pk `id`, fk `category_id` → `product_categories` cascade delete, `language_tag`, `name`, `description?`, `seo_title?`, `seo_description?`, timestamps). Unique index on `(category_id, language_tag)`. TypeID prefix `pctr`.
- `product_tag_translations` (pk `id`, fk `tag_id` → `product_tags` cascade delete, `language_tag`, `name`, timestamps). Slimmer than category translations — tags are short labels, no description / SEO blurbs (per #502 non-goals). Unique index on `(tag_id, language_tag)`. TypeID prefix `pttr`.

Service + admin routes (mirroring destinations translations):

- `GET /v1/admin/products/product-category-translations` — list with `categoryId` / `languageTag` filters + pagination
- `POST /v1/admin/products/product-categories/:id/translations` — upsert (replaces row if `(categoryId, languageTag)` exists)
- `PATCH /v1/admin/products/product-category-translations/:id`
- `DELETE /v1/admin/products/product-category-translations/:id`
- Same shape for product tags under `/product-tag-translations`.

Catalog policy + projection:

- `productTaxonomyCatalogPolicy` flips `categories[]`, `primaryCategoryName`, `tagLabels[]` to `localized: true` + `reindex: "entry-locale"`. Other fields (`categoryIds[]`, `categorySlugs[]`, `primaryCategoryId`, `primaryCategorySlug`, `tagIds[]`) stay single-locale — IDs are stable, and slugs stay canonical because operators want stable URLs that don't shift when translations are edited (per #502 non-goals).
- `createProductTaxonomyProjectionExtension` fetches translations for `slice.locale` (one query per entity type) and overrides the canonical `name` per row when an entry exists. Missing entries fall back to the canonical column.
- Primary tie-break still uses the canonical row name so the picked primary stays stable across locales — only the rendered string changes.

**Backfill explicitly skipped.** The projection's fallback to `productCategories.name` / `productTags.name` already preserves behavior on upgrade — operators who never create translations see the canonical English name on every locale slice, exactly as before. A SQL backfill seeding `en-GB` rows would either require non-strict TypeIDs (breaking `decodeId()` for those rows) or a separate Node seeding step. Both costs aren't justified when behavior is preserved by the fallback. Operators who want pre-populated `en-GB` translations to edit through admin can call the upsert endpoint per category (a one-off script). Reviewers who disagree, push back and we'll add the backfill.
