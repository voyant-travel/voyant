# i18n

## Goals

The i18n system is designed around five rules:

- Shared runtime behavior lives in `packages/i18n`.
- Reusable UI defaults live in the owning `*-ui` package, not in app templates.
- Admin app bundles compose package-owned definitions and add only real app-specific overrides.
- Dynamic client-specific changes use one override shape.
- Tests cover runtime behavior and composition, not translation prose.

## Ownership

Use this ownership model when adding or changing messages:

- Reusable UI surface shipped by a package:
  put defaults in `packages/<module>-ui/src/i18n/<locale>.ts`
- Shared admin surface used by multiple apps:
  put defaults in `packages/i18n/src/admin/<module>.ts`
- App-specific admin surface:
  put defaults in `packages/i18n/src/admin/<module>-operator.ts` or `packages/i18n/src/admin/<module>-dmc.ts`
- App-level composed bundle:
  put it in `packages/i18n/src/admin/app-operator.ts` or `packages/i18n/src/admin/app-dmc.ts`
- Template wrapper:
  keep only locale resolution, app-specific overrides, and typed `useAdminMessages()`

The template files should not own default dictionaries or large message contracts. Registry copies should consume package-owned definitions where a matching `*-ui` package exists.

## Naming

Use these file naming conventions:

- Reusable UI locale files: `en.ts`, `ro.ts`, and `messages.ts` inside `src/i18n/`
- Shared module defaults: `<module>.ts`
- App-specific module defaults: `<module>-operator.ts`, `<module>-dmc.ts`
- App bundle files: `app-operator.ts`, `app-dmc.ts`
- Navigation/core bundles: `operator-nav.ts`, `dmc-nav.ts`, `core-operator.ts`, `core-dmc.ts`

Use these export naming conventions:

- Message definitions: `<name>Messages` or `<name>MessageDefinitions`
- Message contract types: `<Name>Messages`
- Locale entrypoints: `<name>En`, `<name>Ro`

Examples:

- `adminBookingsMessages`
- `OperatorAdminMessages`
- `dmcAdminMessageDefinitions`
- `cruisesUiMessageDefinitions`
- `cruisesUiEn`

## Override model

There is one canonical dynamic override shape:

```ts
type LocaleMessageOverrides<T> = {
  shared?: DeepPartial<T> | null
  locales?: Partial<Record<string, DeepPartial<T>>> | null
}
```

The canonical app-side storage contract is:

```ts
uiPrefs.i18n.admin
```

If another source is introduced, such as organization settings or CMS-managed text overrides, normalize it into the same `LocaleMessageOverrides<T>` shape before passing it into the runtime.

Do not create parallel override formats per app or per feature.

## Resolution order

The runtime resolves messages in this order:

1. fallback locale definitions, usually `en`
2. resolved locale definitions, for example `ro` from `ro-RO`
3. shared overrides
4. locale-specific overrides

This logic lives in `packages/i18n/src/runtime.tsx`.

For reusable packages, use the package helpers exported from `packages/i18n/src/package-runtime.tsx` to create isolated provider contexts. Do not reuse one global messages context across unrelated packages.

## Authoring workflow

When adding a new UI string:

1. Find the package that owns the UI surface.
2. If it is reusable UI, add the key to the owning `packages/<module>-ui/src/i18n/messages.ts` contract.
3. Add the same key to every supported locale for that package.
4. If it is admin-only shared copy, add it to the correct `packages/i18n/src/admin/*.ts` module instead.
5. Export the module or app bundle from `packages/i18n/src/index.ts` if needed.
6. Consume it from the app wrapper or component surface.
7. Run `pnpm i18n:check`.
8. Run `pnpm i18n:check:ui-literals` when touching an i18n-enabled package or registry root.
9. Run the relevant typechecks.

When adding a new app-specific override:

1. Prefer adding it in the package-owned app bundle first.
2. Keep starter-local overrides only if they are truly starter-owned.
3. If starter-local overrides grow, move them back into the package bundle.

When adding a new reusable package seam:

1. Add `src/i18n/messages.ts`, `en.ts`, `ro.ts`, `provider.tsx`, and `index.ts`.
2. Export per-locale entrypoints from that package's `package.json`.
3. Use `createPackageMessagesContext(...)` to create an isolated provider.
4. Prefer locale-aware formatters from `createLocaleFormatters(...)` or the package `use...I18n()` hook over host-default `toLocaleString()` calls.

## Testing

Do test:

- locale fallback behavior
- deep override merge behavior
- isolated package context behavior
- locale-aware formatter behavior
- `uiPrefs.i18n.admin` parsing
- app bundle composition smoke checks

Do not test:

- every translation value
- every key in every dictionary
- giant locale snapshots

The current minimal i18n contract tests live in `packages/admin/tests/unit/i18n.test.tsx`.

## Parity check

Run the locale parity checker before merging i18n changes:

```sh
pnpm i18n:check
```

This validates that every non-`en` locale matches the English shape for admin bundles and any package-owned definition sets that have opted in.

Run the hardcoded string scan for migrated package surfaces:

```sh
pnpm i18n:check:ui-literals
```

This scan is opt-in. It only checks packages and registry roots that expose `i18n/index.ts`, so rollouts can happen incrementally.
