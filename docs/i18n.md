# Internationalization

Voyant treats locale, time zone, messages, and formatting as one application
contract. A localized surface must not depend on browser defaults for any
user-visible text, number, currency, date, time, plural, or list.

## System guarantees

- Reusable UI copy is owned by the package that renders it.
- English is the required fallback locale.
- Locale resolution supports language and regional layers (`ro`, then
  `ro-RO`).
- ICU MessageFormat handles interpolation, plural, select, number, date, and
  time messages with CLDR rules.
- Locale and time zone travel together through every package provider.
- Authenticated account preferences outrank device cache and browser defaults.
- Preference writes are optimistic, persisted through the auth profile, and
  rolled back when persistence fails.
- Server-provided preferences are the initial authenticated render, avoiding a
  device-locale flash during hydration.
- Dynamic overrides are schema-sanitized, prototype-safe, ICU-validated, and
  cannot introduce unknown keys or change required ICU arguments.
- Locale catalogs and ICU arguments are checked for parity in CI.
- Localized roots are AST-scanned for hardcoded UI copy and ambient locale
  formatting.

## Ownership

Put a message in the deepest package that owns the rendered surface:

- Reusable package UI:
  `packages/<module>-react/src/i18n/`
- A package sub-surface with an independent provider:
  `packages/<module>-react/src/<surface>/i18n/`
- Shared admin shell or application composition:
  `packages/i18n/src/admin/`
- Operator- or DMC-specific admin composition:
  `packages/i18n/src/admin/*-operator.ts` or `*-dmc.ts`
- Application bundles:
  `packages/i18n/src/admin/app-operator.ts` and `app-dmc.ts`

Host applications own wiring, not package defaults. A host may choose locale,
time zone, account persistence, and client overrides; it should not duplicate a
package catalog.

Setup/onboarding copy follows the same rule. A setup step contributed by
`inventory-react`, for example, is defined and translated by
`inventory-react`, even when the admin shell renders the step.

## Package interface

An i18n-enabled package normally exposes:

- `messages.ts`: the message contract
- `en.ts`: required fallback catalog
- `ro.ts`: Romanian catalog
- `provider.tsx`: isolated package provider and hooks
- `index.ts`: the public i18n entrypoint

Use `createPackageMessagesContext(...)` from `@voyant-travel/i18n`. Every
provider accepts `locale`, `timeZone`, and optional typed `overrides`.

Components should use the package `use...I18n()` or
`use...I18nOrDefault()` hook when they render formatted values:

```tsx
const { formatCurrency, formatDateTime, formatMessage, messages } =
  useBookingsUiI18nOrDefault()

return (
  <>
    <span>{formatCurrency(totalCents / 100, currency)}</span>
    <time>{formatDateTime(createdAt)}</time>
    <p>{formatMessage(messages.paymentCount, { count })}</p>
  </>
)
```

Use the messages-only hook only when the component renders no localized values.

## Locale and time-zone authority

`LocaleProvider` has two authority modes:

- `device`: local storage, then configured default, then browser locale/time
  zone, then the fallback locale.
- `account`: the authenticated profile is authoritative. The device cache is
  refreshed from the account and never overrides it.

The authenticated admin workspace uses account authority. Locale and time-zone
changes call the profile update port, update the UI optimistically, and roll
back both state and cache if the server rejects the write.

Nested locale providers are supported. The deepest mounted provider owns
`document.documentElement.lang`; unmounting it restores the next active
provider deterministically.

## Message resolution

For a requested locale such as `ro-RO`, resolution is:

1. fallback catalog (`en`)
2. language catalog (`ro`)
3. exact regional catalog (`ro-RO`)
4. shared overrides
5. language override (`ro`)
6. exact regional override (`ro-RO`)

Missing or invalid locale tags safely fall back to English. Locale keys are
canonicalized, so `ro_RO` and `ro-RO` resolve consistently.

## ICU messages

Use ICU MessageFormat for grammatical choices and interpolation:

```ts
paymentCount:
  "{count, plural, =0 {No payments} one {# payment} other {# payments}}"
```

Do not assemble translated sentences with string concatenation. Keep the whole
sentence in the catalog so translators can change word order.

Romanian plurals must include the CLDR categories required by the sentence,
commonly `one`, `few`, and `other`.

Static parity checks reject invalid ICU syntax and argument drift between
locales. Dynamic overrides are rejected when their ICU syntax is invalid or
their argument names/types differ from the fallback message.

## Dynamic overrides

The canonical override shape is:

```ts
type LocaleMessageOverrides<T> = {
  shared?: DeepPartial<T> | null
  locales?: Partial<Record<string, DeepPartial<T>>> | null
}
```

The admin application bundle reads this shape from:

```ts
uiPrefs.i18n.admin
```

Packages may receive the same typed shape through their provider. Any future
organization, CMS, or client-specific source must normalize to this contract.
Do not introduce feature-specific override formats.

Untrusted preference JSON must pass through
`getLocaleMessageOverridesFromUiPrefs(...)`. Direct provider overrides are also
sanitized defensively by the resolver.

## Formatting rules

Never use ambient formatting in localized UI:

```ts
// Incorrect: changes with the device, not the selected app locale.
value.toLocaleString()
new Intl.DateTimeFormat(undefined).format(date)
```

Use bound package formatters, or pass an explicit locale when a pure helper
cannot access the provider:

```ts
formatNumber(value)
formatDateTime(date)
new Intl.DisplayNames(locale, { type: "region" })
```

Use the configured time zone for user-visible instants. Date-only values are
normalized to avoid shifting the calendar day across time zones.

`i18n-format-ok` is reserved for non-rendering locale-insensitive operations,
such as calling `Intl.DateTimeFormat().resolvedOptions().timeZone` to discover
the browser time zone. Add a comment explaining why the exemption is safe.

## Route and lazy-loading boundaries

Lazy admin routes declare a package-owned messages provider. The admin binder
passes both locale and time zone through the lazy boundary, so route chunks do
not need to import the central application catalog.

Prefer route-local providers for new package UI. The central operator/DMC
bundles remain the composition seam for the shell and legacy shared admin
surfaces, not a destination for new package-owned copy.

## Authoring workflow

When adding a localized surface:

1. Identify the package that renders it.
2. Add or extend that package's message contract.
3. Add the key to every supported locale.
4. Use ICU for interpolation, plurals, and selects.
5. Consume the package provider hook.
6. Use its bound number/date/currency/message formatters.
7. Ensure lazy route/provider wiring forwards locale and time zone.
8. Run `pnpm i18n:check`.
9. Run the relevant package tests and typecheck lane.

The repository's primary CI workflow runs these checks on Depot in
`.depot/workflows/ci.yml`.

## Enforcement

`pnpm i18n:check` runs three fail-closed gates:

- Admin parity: validates every composed admin definition set.
- Package parity: recursively discovers package definition exports and validates
  catalog shape, arrays, ICU syntax, and ICU arguments.
- UI AST scan: discovers every package with an i18n seam and rejects hardcoded
  user-facing JSX/property copy plus ambient `Intl`/`toLocale...` formatting.

The gates fail when discovery unexpectedly returns no catalogs or no localized
roots, preventing a configuration mistake from silently disabling enforcement.

## Testing

Test behavior and contracts:

- fallback, language, and regional layering
- shared/language/regional overrides
- malformed and malicious override rejection
- ICU plural behavior and missing arguments
- locale/time-zone formatting
- account preference precedence, persistence, and rollback
- hydration-safe initialization
- nested provider document-language ownership
- lazy route provider propagation

Avoid giant translation snapshots or tests that merely repeat prose. Parity and
schema checks cover completeness; focused tests should cover runtime behavior.
